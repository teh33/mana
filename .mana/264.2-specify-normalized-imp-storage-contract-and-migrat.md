  Organize imp files by semantics first, not by individual feature buckets.
  Primary axes:
  1. scope — user-global vs project-local
  2. authorship — user-authored vs imp-generated
  3. durability — precious history vs rebuildable cache
  4. sensitivity — secret vs non-secret

  Proposed conceptual buckets:
  - user config/content: user-authored, editable, syncable setup and extension content
  - user data: durable generated history/value the user likely wants to keep
  - user state: machine-local mutable runtime state that imp updates
  - user secrets: secure-store only, never mixed with ordinary files
  - project-local `.imp/`: project behavior/overrides/extensions only

  Draft topology:
  - user config/content root:
    - `config.toml`
    - `soul.md`
    - `memory.md`
    - `user.md`

    - `auth.json` metadata

Goal: define the normalized imp storage contract another worker can implement cold.

Current state:
- Storage paths are currently implicit, split by callsite, and partly platform-special-cased.
- The repo needs a durable decision on how imp distinguishes user config, user data, project-local config, project-local extension content, caches/indexes, and compatibility lookups.

Steps:
1. Propose a normalized topology for durable imp files, explicitly separating:
   - user-global config/state intended to sync or edit (`config.toml`, `soul.md`, skills, prompts, imported AGENTS)
   - user-global durable data (`sessions`, session index, perhaps usage/report stores if applicable)
   - user-global secrets/auth metadata boundaries
   - project-local `.imp/` content (config, soul, prompts, skills, lua, tools)
   - ephemeral cache/runtime data if any should be distinct
2. Define platform behavior clearly, including whether imp should use an internal XDG-like contract everywhere or adopt native macOS dirs for data/config in a more systematic way.
3. Define precedence and compatibility lookup rules so legacy paths can still be discovered during migration.
4. Decide which path rules belong in imp-core versus docs only.
5. Produce a worker-ready spec with explicit target helpers/APIs and non-goals.

Done when:
- a single normalized storage contract exists for current imp reality and near-term migration
- the contract is explicit enough to drive implementation without re-deciding architecture during coding

Out of scope:
- implementing the refactor
- deleting legacy data immediately
- changing mana ownership boundaries

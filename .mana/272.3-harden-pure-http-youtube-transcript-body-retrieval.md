Follow-up hardening for YouTube transcript body retrieval. Current state from 272.1 implementation: watch-page `ytInitialPlayerResponse` is extractable and caption track metadata exists for sample `McO_xcf4IYw`, but the watch-page caption `baseUrl` returns HTTP 200 with an empty body in this environment. `yt-dlp` obtains working caption URLs from an `android_vr` client, while naive `/youtubei/v1/player` WEB/ANDROID_VR requests either lack captions or return `LOGIN_REQUIRED` / bot confirmation. Goal: make the pure Rust/HTTP extractor robust enough to retrieve transcript bodies without yt-dlp, or produce principled diagnostics if YouTube requires unavailable auth/PoToken/client context. Steps: inspect current yt-dlp YouTube client profiles/request payloads and headers; identify the minimal unauthenticated client context that returns signed caption URLs for public videos; implement it in `imp/crates/imp-core/src/tools/web/youtube.rs` without adding yt-dlp; add offline tests for client-profile response parsing; add a network ignored smoke test for sample URL. Scope boundaries: no media download, no shelling out, no heavy dependency without approval, no cookies by default.

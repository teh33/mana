## Task
Finish the verify-dedup integration that 27.3 intentionally left as a pool-side seam.

## Current state
- `mana/crates/mana-pool/src/types.rs` now has `DispatchUnit.verify_command: Option<String>` and `VerifyGroup`.
- `mana/crates/mana-pool/src/dispatch.rs` now has grouping helpers/tests and emits `PoolEvent::VerifyGroupRun`.
- The active run path still lives in `mana/crates/mana-cli/src/commands/run/mod.rs` and currently resolves deferred verify by calling `mana_core::ops::batch_verify::batch_verify(mana_dir)` after agents complete.
- Current pool callers do not yet populate `DispatchUnit.verify_command`, so the new pool seam is not yet exercised end-to-end.

## Goal
Choose and implement one ownership path so deferred verify dedup is not split ambiguously between `mana-cli` and `mana-pool`.

## Required approach
1. Inspect how `SizedUnit` / run planning data carries verify commands today in `mana/crates/mana-cli/src/commands/run/plan.rs`, `ready_queue.rs`, and any pool adapter code.
2. Populate `DispatchUnit.verify_command` from the existing unit verify field when constructing pool dispatch inputs.
3. Decide the single source of truth for deferred verify execution:
   - either let `mana-pool` own grouped execution and have CLI consume pool outcomes/events,
   - or keep `mana-core::ops::batch_verify` as the execution owner and reduce `mana-pool` to grouping/event surfaces only.
4. Remove duplicate/conflicting behavior so only one path actually runs deferred verify commands.
5. Keep `PoolEvent::VerifyGroupRun` visible in the chosen path.
6. Add or update focused tests around the chosen integration point.

## Files
- `mana/crates/mana-cli/src/commands/run/mod.rs` (inspect/modify — current post-agent batch verify owner)
- `mana/crates/mana-cli/src/commands/run/ready_queue.rs` (inspect/modify — current active direct runner)
- `mana/crates/mana-cli/src/commands/run/plan.rs` (inspect if dispatch inputs are built there)
- `mana/crates/mana-pool/src/types.rs` (read/adjust only if integration needs small type cleanup)
- `mana/crates/mana-pool/src/dispatch.rs` (read/adjust only if integration requires event/outcome cleanup)
- `mana/crates/mana-core/src/ops/batch_verify.rs` (inspect — current grouped deferred verify owner)

## Acceptance
- Deferred verify dedup has one clear execution owner.
- `DispatchUnit.verify_command` is populated where pool dispatch inputs are built, or the type is intentionally removed if the chosen owner makes it unnecessary.
- Grouped verify events remain observable.
- No path runs the same deferred verify command twice.
- `cargo check -p mana-pool -p mana-cli` passes.

## Don't
- Do not leave both CLI post-run batch verify and pool-side grouped execution active without an explicit coordination contract.
- Do not broaden into unrelated run-loop refactors.
- Do not rely on chat context; use the file paths and current state above.

  Numbering/identity suggestion from current discussion:

  Best direction: decouple canonical identity from human display path.

  Recommended model:
  - Keep existing `x.y.z` IDs as compatibility aliases during migration.
  - Introduce stable opaque canonical IDs for internal references, e.g. `task_01J...` or deterministic UUID/ULID-style IDs.
  - Add separate human display path fields, e.g. `display_id: 1.a.1`, `parent_id: task_...`, `sort_key: ...`, `slug: ...`.
  - Use `1.a.1` as a display/navigation path, not the database primary key.

  Why:
  - Stable canonical IDs survive renumbering, moves, insertion, sync, and hosted/server workflows.
  - Human display paths stay readable and scannable.
  - `1.a.1` can communicate hierarchy better than `1.1.1`: numeric root/epic, letter phase/section, numeric task/item. Example: `2.b.4` = second epic, phase/track b, fourth task.

  Migration-safe path:
  1. Add optional `uid` and `display_id` fields while preserving existing `id`.
  2. Treat current `id` as both canonical/display until new fields exist.
  3. New units can get `uid` immediately; old units get generated `uid` lazily via `mana doctor fix` or migration command.
  4. CLI accepts both canonical UID and display path aliases.
  5. Dependencies eventually store canonical UID plus display alias for readability.

  Open decision: whether to make `1.a.1` generated from parent/order metadata or manually stable once assigned. Recommendation: generated initially, but preserve aliases after renumbering so old references keep resolving.

Goal: Design and later implement mana's identity split: stable canonical UID slug plus movable human display `mana_id`.

Settled decision:
- Add a stable canonical UID/slug for durable internal identity.
- Add separate human-facing `mana_id`/display path for navigation, e.g. `1.a.1`.
- `mana_id` should automatically rename/renumber when units move, are reparented, or hierarchy/order changes.
- Old `mana_id` values must be retained as aliases so existing links, dependency references, chat references, and history keep resolving.
- Durable references and dependencies should eventually store UID, not mana_id.
- Filenames may include `mana_id` + slug for readability, but lookup/identity must not depend on filename stability.

Recommended data model:
```yaml
uid: task_01JABCDEF...
mana_id: 1.a.1
aliases:
  - 1.2.3
slug: add-sqlite-index
parent_uid: task_01JAAAA...
sort_key: ...
```

Terminology:
- `uid`: canonical durable identity. Stable across moves, renumbering, sync, hosted workflows, and filename changes.
- `mana_id`: current human display/navigation path. Mutable and auto-renumbered to reflect hierarchy.
- `aliases`: old `mana_id` values and possibly legacy `x.y.z` IDs accepted for lookup compatibility.
- `slug`: human-readable name component, not identity.
- `parent_uid`: canonical parent relation.
- `sort_key`: ordering/hierarchy key used to generate `mana_id`.

Lookup rules:
1. CLI/API accepts UID, current `mana_id`, and aliases.
2. UID match wins over display/alias match.
3. Ambiguous aliases must produce an explicit disambiguation error.
4. Dependencies should migrate to UID storage, optionally with display alias retained for human readability.
5. Old `x.y.z` IDs remain valid aliases during migration.

Move/renumber rules:
1. Moving/reparenting a unit updates `parent_uid`, `sort_key`, and generated `mana_id`.
2. Previous `mana_id` is appended to `aliases` if not already present.
3. Child `mana_id`s are regenerated recursively if display path derives from parent hierarchy.
4. UID and dependency identity do not change.
5. Any filename rename is secondary and should happen only after UID/alias lookup is working.

Implementation decomposition:
1. Inspect current ID parsing, lookup, filename, parent/dependency, archive, and SQLite index assumptions.
2. Specify UID format (ULID-like `task_...`, UUID, deterministic ID, etc.) and migration behavior.
3. Add optional fields to unit schema: `uid`, `mana_id`, `aliases`, `parent_uid`, `sort_key` while preserving current `id`.
4. Extend lookup to resolve UID/current mana_id/aliases with ambiguity handling.
5. Add migration/doctor support to backfill UIDs and aliases for existing units.
6. Update dependency/reference storage toward UID while preserving display aliases.
7. Implement move/reparent renumbering of `mana_id` and alias retention.
8. Only then consider filename renames or replacing visible CLI IDs.

Out of scope for first implementation slice:
- Breaking current `x.y.z` references.
- Renaming all files immediately.
- Hosted/server identity semantics beyond keeping UID stable.
- Changing SQLite canonical/derived boundary.

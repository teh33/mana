Goal: Evaluate improvements to mana's numbering/identity scheme separately from storage backend work.

Current discussion context:
- User noted current `x.y.z` numbering may be improvable and suggested something like `x.a.y` or another scheme.
- This may be out of scope for storage, but identity affects storage schema, sync, merges, references, and human ergonomics.

Steps:
1. Inspect current ID assumptions enough to ground the design: lookup, parent/child, slug filenames, archive references, dependencies, and display in CLI.
2. Compare options: keep hierarchical numeric IDs; introduce stable opaque object IDs plus human display paths; use mixed path segments like `x.a.y`; use parent/order metadata with generated display IDs.
3. Evaluate each for: merge conflicts, insertion/reordering, long-lived references, URL/API stability, human scanability, and migration cost.
4. Recommend whether storage evolution should decouple canonical identity from display numbering.
5. Define compatibility rules for existing IDs and references.

Files:
- `docs/architecture/mana-id-numbering-evolution.md` (create)
- relevant mana lookup/graph code (read only)

In scope:
- Architecture/design only.
- Migration-safe recommendation.

Out of scope:
- Renumbering existing mana units.
- Implementing ID migration.

Do not:
- Break existing `x.y.z` references.
- Conflate canonical object identity with display ordering unless explicitly justified.

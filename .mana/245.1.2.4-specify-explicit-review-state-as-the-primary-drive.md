## Task
Write a root architecture doc for representing explicit review requirements/state in mana as the primary driver of future human-only close.

## Why this unit exists
The current discussion settled an important policy direction:
- future human-only close should be driven primarily by explicit review
- this should not be treated as an automated heuristic-driven use-case
- blockers and later blocker-based gating handle earlier lifecycle clarification/gating responsibilities, while human-only close remains a distinct end-of-lifecycle review/signoff surface

## What the doc must specify
1. What explicit review means in the mana model
   - requirement/policy
   - current review state
   - review outcome
2. How explicit review remains distinct from blockers
3. How explicit review should participate in future close-time policy
4. Whether review requirements live on units directly, in project config, or as a layered combination
5. What role heuristics may still play, if any, without becoming the primary driver
6. How this representation can coexist with current `feature: true` compatibility behavior during migration

- do not require this representation to land in the first blocker slice

## Task
Create a root design/decomposition unit for the policy layer of the file-native memory architecture. This unit should capture retrieval, compaction, conflict, staleness, and failure-memory promotion rules in a form suitable for later implementation work.

## Policy areas to define
1. Compaction / promotion
   - when to emit checkpoints (handoff, retry, blockage, verified milestone, completion, context pressure)
   - when to promote repeated or costly failures into lessons
   - when to promote ambiguous/conflicting states into decisions
2. Retrieval cascade
   - default order: bootstrap packet → latest relevant checkpoint → relevant lessons/runbooks → accepted claims → raw evidence on demand
   - ranking dimensions: scope match, path/entity/task match, recency, evidence strength, unresolved status, failure-signature match, freshness penalty, redundancy penalty
3. Conflict representation
   - claims as first-class propositions with statuses such as `candidate`, `supported`, `contested`, `accepted`, `superseded`, `retracted`, `needs_revalidation`
   - contested claims remain durable; resolution occurs via new adjudicating objects rather than rewriting history
4. Staleness
   - how evidence becomes stale (commit/path changes, env changes, review windows)
   - how claim status/confidence degrades when supporting evidence becomes stale
5. Failure memory
   - raw failed attempts remain durable
   - reusable lessons are promoted abstractions with scope, supports, counterexamples, retry guidance, and invalidation/deprecation behavior

## Why root scope
This policy affects both mana durable state and imp context assembly/retry behavior, so it belongs in the root graph.

## Do not
- do not make raw attempt logs the default prompt input
- do not hide conflicts by overwriting old claims
- do not promote every failure into permanent memory

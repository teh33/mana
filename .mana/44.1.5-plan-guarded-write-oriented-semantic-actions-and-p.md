## Goal
Plan the later phase for mutating semantic actions in `imp`.

## Current state
- `docs/design/code-intelligence-strategy.md` establishes that write-oriented semantic actions belong later than read-oriented queries and require stricter runtime policy.
- `docs/design/imp-write-oriented-code-intelligence-policy.md` now captures the durable planning decisions for this slice.
- The key architectural distinction is explicit: semantic writes are a separate `imp` capability, not general script execution and not ordinary text edits.

## Scope
Planning only. No implementation.

## Durable decisions captured here
1. Enable `rename`, `organize imports`, and selected `code actions` only after read-oriented semantic inspection exists for the same backend.
2. Require preview before apply, automatic checkpoint before mutation, explicit approval for risky delegated rewrites, and deterministic post-action verify.
3. Treat opaque command-backed code actions as disabled in the initial phase.
4. Keep backend/language risk explicit; do not assume all backends support safe mutation equally.
5. Keep semantic writes distinct from both plain text edits and the general script sandbox.

## Deliverable
- `docs/design/imp-write-oriented-code-intelligence-policy.md`

## Follow-on decomposition
- `44.1.2.1` should define the typed policy/config model and capability matrix that turns this planning note into an implementable runtime contract.

## Constraints
- Do not broaden into generic extension policy.
- Do not collapse these tools into the general script sandbox.
- Do not assume uniform safety across languages/backends.

## Done
Done when the design note exists and the durable plan is represented in root mana so later workers can continue from the graph instead of reconstructing it from chat.

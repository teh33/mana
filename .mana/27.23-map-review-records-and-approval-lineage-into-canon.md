Goal: finish the review/approval half of scheduler-facing autonomy gating now that 27.17 added durable `approval` / `review` / `verify` state on `Unit.autonomy_disposition` and close-path verify normalization.

Current state:
- `mana-core` now persists `approval: ApprovalState`, `review: ReviewState`, and `verify: VerifyPosture` on `Unit.autonomy_disposition`.
- `mana-core` close flow now durably normalizes deferred verify, verify failure, and verify-frozen mismatch into that disposition.
- `mana-review` persists actual review records in `.mana/reviews/*.yaml` and already depends on `mana-core`.
- `mana-core` also owns durable `ApprovalRecord`, `PromotionRecord`, and `ReviewGateOutcome`, but current repo reality does not yet map those artifacts/lineage into canonical scheduler-facing `review` / `approval` state.
- Pool-side keep-going remains invalid until this mapping exists upstream and schedulers consume only the resulting disposition.

Task:
Implement the canonical mana-core mapping from durable review/approval lineage into `Unit.autonomy_disposition` without creating a `mana-core -> mana-review` crate cycle.

Concrete steps:
1. Inspect `mana/crates/mana-review/src/state.rs` and current review file shape in `.mana/reviews/` to confirm what mana-core can safely read as file-native durable state without depending on mana-review crate types.
2. Inspect `mana/crates/mana-core/src/unit/types.rs` for `ApprovalRecord`, `PromotionRecord`, and `ReviewGateOutcome` and decide the minimal canonical evaluator inputs.
3. Add a mana-core-owned mapping layer that classifies scheduler-facing `review` / `approval` state from durable lineage, including at minimum:
   - review not required / required pending / approved / rejected / unknown
   - approval not required / required pending / approved / rejected / unknown
4. Ensure the mapping distinguishes:
   - review gate outcome (`skipped` / `optional` / `mandatory`)
   - actual skeptical review result when present
   - approval-layer authorization when present
5. Update the canonical `Unit::refresh_autonomy_disposition()` path to consume this mapping rather than only carrying forward prior state.
6. Add focused tests covering:
   - mandatory review with no review lineage -> review/approval remain blocking
   - persisted approved review -> review satisfied but approval still separate when absent
   - approval record present -> approval satisfied
   - rejected review / denied approval -> explicit blocking state
7. Document any remaining repo-reality gaps that still prevent full automatic classification.

Files:
- `mana/crates/mana-core/src/unit/mod.rs` (modify)
- `mana/crates/mana-core/src/unit/types.rs` (read / modify only if ownership requires it)
- `mana/crates/mana-review/src/state.rs` (read for file-native shape only; avoid creating a crate cycle)
- nearby mana-core tests (modify)

Do not:
- Do not import `mana-review` enums/types into `mana-core` if that creates or implies a crate cycle.
- Do not let mana-pool parse review artifacts directly.
- Do not collapse review result and approval result into one field.
- Do not add pool-side behavior yet.

This unit exists to externalize the remaining durable plan from 27.17: review/approval state is now modeled on the scheduler-facing summary, but it still needs canonical population from durable lineage before pool-side keep-going can ever be valid.

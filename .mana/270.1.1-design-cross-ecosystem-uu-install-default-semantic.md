Design the generalized non-Cargo semantics for `uu install --default`. Current state: Cargo projects are handled natively in uu, but non-Cargo projects still rely on repo-local post-install hooks when present, and otherwise `--default` is a no-op after the native install step. This unit is to define the durable contract and the first extension targets so the feature stops being Cargo-special and becomes a coherent cross-ecosystem capability.

Required design output:
1. Define what `--default` promises across ecosystems: after install, the user-visible command resolved by the shell should point at the fresh install, using user-local paths only by default.
2. For each target ecosystem, specify:
   - how uu infers or is told the command name(s)
   - where the installed artifact is expected to live after the ecosystem-native install step
   - how uu safely updates the default command path without requiring sudo or mutating PATH
3. Start with practical next ecosystems where this is tractable and common, likely Node package bins and Python tool/script installs.
4. Explicitly classify unsupported or ambiguous ecosystems where native `--default` should error or remain unsupported rather than guessing.
5. Recommend whether repo hooks should remain as an escape hatch after native support grows, and under what constraints.

Keep this as a design/spec unit, not broad implementation. Ground it in the current uu install architecture and the just-landed Cargo defaulting behavior.

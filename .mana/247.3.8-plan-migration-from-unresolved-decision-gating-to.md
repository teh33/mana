## Task
Write a root follow-on architecture doc for migrating mana's current unresolved-decision dispatch gate to blocker-based gating after the first blocker slice lands.

## Grounded current state
Current repo inspection in this session showed:
- `Unit.decisions` currently stores unresolved questions that block autonomous execution
- `mana-core/src/ops/run.rs` has `blocked_unit_for_unresolved_decisions()`
- `compute_ready_queue()` uses unresolved `decisions` as a canonical blocked reason (`unresolved_decision`)
- `mana-cli/src/commands/run/plan.rs` and `run/mod.rs` surface and optionally override unresolved-decision blocking during dispatch planning
- `feature: true` / `CloseOutcome::FeatureRequiresHuman` is a separate close-time human-signoff mechanism in `close.rs`

## Why this unit exists
The blocker semantics settled in this conversation changed the model:
- blockers absorb unresolved blocking questions and impediments
- decisions now mean resolved/adopted outcomes and rationale
That means the current unresolved-decision dispatch gate is conceptually obsolete and should eventually migrate to blockers, but this should be a later policy migration, not part of the first blocker slice.

## What the doc must specify
1. Which current run-planning and context surfaces rely on unresolved `decisions`
2. How blocker-based gating should replace unresolved-decision gating later
3. Whether all open blockers gate dispatch or only selected blocker kinds/statuses
4. How blocked reason rendering/messages should change
5. How the migration preserves the separate feature human-close policy unless explicitly redesigned
6. How mixed-state repos behave during transition

Goal: define the first backend-neutral session-host abstraction for imp so persistent terminal hosting can be supported cleanly, starting with tmux as a possible backend but without baking tmux semantics into the generic bash tool.

Current state:
- Root work already covers detached/background local execution (`50.25`, `50.26`) and an optional tmux host-side session surface (`50.27`).
- The planning conversation refined the architecture decision: tmux should supplement bash, not replace it; raw `tmux ...` commands may remain available through bash as an escape hatch, but supported persistent terminal hosting should not be modeled as a bash option.
- A missing durable artifact is the actual backend-neutral `session_host` decomposition that another worker could inherit cold.

Steps:
1. Define why first-class persistent terminal hosting is the wrong fit for the generic `bash` tool even if `bash` can still shell out to tmux.
2. Specify the minimum `session_host` contract for imp-local detached execution, including:
   - session identity
   - create/list/show
   - exec/send input
   - capture output / transcript slices
   - attach/reattach / takeover metadata
   - terminate / cleanup
3. Describe how detached/background imp run identity should map onto session-host instances without making the host the durable source of truth.
4. Define the initial backend strategy:
   - tmux as an optional first backend
   - room for zellij/native supervisor/terminal-emulator integrations later
5. Clarify ownership split:
   - bash stays generic command execution
   - session_host owns persistent terminal/session lifecycle
   - imp owns runtime semantics and operator UX
   - mana owns durable task state, evidence, and resumable handoff
6. Capture non-goals and guardrails so future work does not let pane history become canonical state.

Relevant references:
- root mana units `50.25`, `50.26`, `50.27`, `248.17`
- docs/rebuild/imp-cli-affordance-sequence.md
- docs/rebuild/imp-detached-execution-cli.md once present
- docs/rebuild/imp-detached-execution-model.md once present

In scope:
- architecture/spec for a backend-neutral session-host surface
- tmux as initial backend framing
- relationship between session hosting and detached/background execution

Out of scope:
- implementing the tool now
- committing to tmux as the only backend
- moving normal shell command execution out of bash

Do not:
- hide first-class tmux support inside bash parameters
- treat the session host as durable memory
- replace detached-run identity with raw pane/session names

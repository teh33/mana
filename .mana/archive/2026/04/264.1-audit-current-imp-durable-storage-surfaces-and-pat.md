  2026-04-16T06:41:34.792047+00:00
  Source-backed audit findings so far:
  - Core path helpers today are minimal and incomplete: `imp/crates/imp-core/src/config.rs` exposes `Config::user_config_dir()` and `Config::session_dir()`, but not a full storage-path API.
  - User-global config-ish files currently hang off `Config::user_config_dir()`: `config.toml`, `auth.json`, `memory.md`, `user.md`, `soul.md`, imported `AGENTS.md`, skills under `skills/` (including agent-created `skills/agent/...`), prompts under `prompts/`, and Lua extensions under `lua/`.
  - Project-local override/discovery surfaces live under `.imp/`: `config.toml`, `soul.md`, `skills/`, `prompts/`, and Lua extensions under `.imp/lua/`.
  - Raw sessions are stored separately via `Config::session_dir()` and currently resolve to the data root (`~/.local/share/imp/sessions` fallback), not under the user config dir.
  - `imp/crates/imp-core/src/tools/session_search.rs` does not use `Config::session_dir()` or any shared data-root helper for the index; it hardcodes a separate macOS Application Support path (`~/Library/Application Support/imp/session_index.db`) and XDG data path elsewhere.
  - `imp/crates/imp-core/src/session_index.rs` only defines the DB schema/indexing API; I have not yet found a production callsite that automatically indexes sessions on session end or startup. Current search evidence suggests indexing may be effectively manual/unwired today, which explains real-world `No sessions indexed yet` failures even when raw sessions exist.
  - `imp/crates/imp-core/src/resources.rs` has explicit user-global + project-local discovery logic for AGENTS/CLAUDE, soul, skills, and prompts, including cwd ancestry walk for project skills.
  - `imp-lua/src/loader.rs` separately discovers user-global `lua/` and project-local `.imp/lua/` extensions.
  - `imp-cli` and `imp-tui` contain many repeated direct `Config::user_config_dir().join("auth.json")` and `Config::session_dir()` callsites, reinforcing that path policy is duplicated at the edges rather than centralized.
  - Shell-tool loading exists as `tools/shell.rs::load_shell_tools(dir, ...)`, but I have not yet confirmed the production discovery roots/callers in the current build path.

  Tentative mismatch taxonomy:
  1. config-vs-data split exists in part, but only for sessions
  2. session index is outside the shared helper surface and platform-special-cased
  3. many durable artifact paths are assembled ad hoc by callers
  4. automatic session indexing may be missing entirely, so path cleanup alone may not fully solve session recovery/search

  2026-04-16T06:52:09.983889+00:00
  Additional source-backed audit findings:
  - Auth/secrets are intentionally split across two storage classes today. `imp-llm/src/auth.rs` persists credential metadata to `auth.json` in the user config dir, while named secret values for `SecretFields` go through the OS keychain backend (`KEYRING_SERVICE = "imp"`). That split should remain explicit in the normalized topology instead of being treated as one flat file store.
  - Import flows currently target the user config tree: `import_skills()` copies discovered external skills into `<user_config_dir>/skills/*`, and `import_agents_md()` copies the first imported instruction file into `<user_config_dir>/AGENTS.md`.
  - I still cannot find production callsites for two durable storage surfaces that exist in modules/docs/tests:
    1. `resources::discover_prompts()` for prompt template files under user/project prompt dirs
    2. `tools::shell::load_shell_tools()` for TOML-defined shell tools
    At the moment these look like defined storage surfaces without active runtime wiring in the current shipped path.
  - The session index gap is stronger than a mere path mismatch: `session_index.rs` defines the DB/indexing API and `session_search.rs` reads it, but current repo search has not yet surfaced a production lifecycle callsite that actually indexes saved sessions during normal CLI/TUI/session shutdown.

  Working storage taxonomy for the audit table:
  - user-global config documents: `config.toml`, `soul.md`, imported `AGENTS.md`
  - user-global configurable content/extensions: `skills/`, `prompts/`, `lua/`
  - user-global durable user data: `sessions/`, `session_index.db`
  - user-global personal memory docs: `memory.md`, `user.md`
  - user-global auth metadata: `auth.json`
  - user-machine secure secret values: OS keychain entries keyed by `imp` service
  - project-local overrides/extensions: `.imp/config.toml`, `.imp/soul.md`, `.imp/skills/`, `.imp/prompts/`, `.imp/lua/`
  - possible but currently unwired/dead surfaces pending confirmation: prompt templates and shell-tool TOML directories

  Proposed follow-on decomposition from the audit:
  1. separate path normalization from lifecycle wiring issues
  2. create one follow-on for session indexing lifecycle/wiring if no existing unit cleanly owns it
  3. create one follow-on to either wire or retire prompt-template and shell-tool storage surfaces so the normalized topology does not preserve dead baggage by accident

Goal: produce a source-backed audit of every durable imp file surface and where the current code resolves it.

Current state:
- `config.rs` provides `user_config_dir()` and `session_dir()` but there is no single `imp storage paths` abstraction.
- `tools/session_search.rs` uses its own index path logic and special-cases macOS.
- `resources.rs` discovers skills/prompts/soul across user-global and project `.imp/` trees.
- `tools/memory.rs`, `tools/extend.rs`, `tools/web/*`, `imp-lua`, `import.rs`, and TUI flows each rely on overlapping path assumptions.
- We already have evidence that these mismatches affect real operator recovery on this Mac.

Steps:
1. Inspect the current path logic for: config.toml, auth.json, sessions, session index, memory.md, user.md, soul.md, prompts, skills, agent-created skills, shell tools, Lua extensions, imported AGENTS.md, and any other durable imp-managed file surfaced by the current code.
2. Build a table with columns: artifact, ownership/scope (user global, machine local data, project local), current path(s), code owner/helper, precedence/discovery behavior, and known mismatch/risk.
3. Separate current shipped reality from desirable future cleanup.
4. Append the audit to the parent epic and identify the smallest coherent follow-on jobs needed.

Goal: capture and resolve the current mismatch between how imp stores raw sessions, how session_search discovers indexed history, and where operators should look when trying to recover a prior conversation on macOS.

Current state:
- During a live support attempt to recover a lost imp/mana architecture conversation, `session_search` returned `No sessions indexed yet. Session search becomes available after your first conversation.`
- Direct disk inspection found a legacy raw session store at `/Users/asher/.imp/sessions/` with one tiny `test` transcript only.
- Code inspection shows current `Config::session_dir()` uses XDG-style data storage (`~/.local/share/imp/sessions` on macOS unless `XDG_DATA_HOME` is set), while `session_search` uses a separate macOS index path at `~/Library/Application Support/imp/session_index.db`.
- This means recovery can fail in at least three ways: raw sessions absent from the expected directory, index DB missing even when raw sessions exist elsewhere, or operators looking in the wrong store because legacy and current paths coexist.

Steps:
1. Inspect session storage code and identify the exact intended session directory and session index paths across macOS codepaths.
2. Inspect the local machine for all plausible raw session stores and index DB locations, including legacy `~/.imp/sessions`, current XDG-style data dirs, and macOS Application Support index locations.
3. Determine whether this is only a local migration/legacy-path issue, an indexing gap, or a broader product/UX mismatch between saved sessions and searchable sessions.
4. Record the operator-facing recovery path: where to look first for raw transcripts, how to tell whether indexing has ever run, and what evidence confirms a session is actually absent.
5. If the repo already has a suitable planning unit for session browsing/search UX, append the findings there; otherwise leave this unit as the execution container for follow-on fixes.

Files:
- `imp/crates/imp-core/src/config.rs` (read — `Config::session_dir()` and path conventions)
- `imp/crates/imp-core/src/tools/session_search.rs` (read — index DB path and missing-index behavior)
- `imp/crates/imp-core/src/session.rs` (read — raw session creation/list/open behavior)
- local machine paths such as `/Users/asher/.imp/sessions/`, `/Users/asher/.local/share/imp/sessions/`, and `/Users/asher/Library/Application Support/imp/session_index.db` (inspect — establish real state)

In scope:
- diagnosis of session storage/index locations
- durable recording of the recovery path and mismatch
- narrowing whether the lost conversation is actually gone versus merely unindexed

Out of scope:
- broad redesign of session UX
- implementing the final fix unless the diagnosis clearly makes a tiny follow-on change obvious

Do not:
- assume `session_search` absence means the transcript was never saved
- assume legacy `~/.imp/sessions` is the only store
- claim recovery without opening the actual files or confirming their absence

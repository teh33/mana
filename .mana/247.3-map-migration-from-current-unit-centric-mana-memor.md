## Task
Write a root migration-gap document at `docs/architecture/memory-migration-gap-map.md` that compares the current mana memory model to the target object-centric file-native memory architecture.

## Why this unit exists
The current discussion identified a durable architectural gap: mana today is primarily unit-centric, with most memory embedded inside mutable unit files, while the target architecture makes immutable durable objects the canonical truth and treats current state as derived/materialized.

## Grounded current-state facts to capture
Use the current repo state already inspected in this discussion:
- canonical durable record today is the `.mana/{id}-{slug}.md` Unit
- `Unit` currently embeds fields like `description`, `acceptance`, `notes`, `decisions`, `attempt_log`, `history`, `paths`, `outputs`, `stale_after`, and `autonomy_disposition`
- `.mana/index.yaml` is a derived/rebuildable cache
- facts already exist as a special unit kind with verify + TTL/staleness
- `memory_context` retrieval is currently unit/fact-oriented (`warnings`, `working_on`, `relevant_facts`, `recent_work`)
- typed attempt/result-scoped observations exist in limited form (`AutonomyObservation` on `AttemptRecord`/`RunRecord`), but there is no general first-class claim/evidence object graph yet

## What the document must explain
1. `## Current mana model`
   - summarize today's unit-centric shape
2. `## Target memory model`
   - summarize object-centric truth with derived current state
3. `## Key gaps`
   - unit-as-container vs object-graph-as-truth
   - facts vs claims+evidence
   - raw attempts vs checkpoints
   - notes/history vs promoted lessons
   - side-effect history vs receipts
4. `## What can stay`
   - unit graph, verify gates, rebuildable indexes, fact/staleness ideas, current typed durability work that can seed the new model
5. `## Migration sequence`
   - suggest an incremental path that preserves current workflows while introducing first-class durable objects and derived views

## Task
Write a root architecture doc for evolving mana's current feature-based human-close gate into a better heuristic/policy model.

## Grounded current state
Current repo inspection in this session showed:
- `close.rs` currently returns `CloseOutcome::FeatureRequiresHuman` when `unit.feature` is true unless forced in an interactive context
- this is a coarse close-time human-review policy keyed off a boolean
- blocker planning in this conversation established that blockers absorb unresolved blocking questions/impediments and later should take over unresolved-decision dispatch gating

## Why this unit exists
The current discussion clarified an important design shift: once blockers handle more human-in-the-loop responsibility earlier in the lifecycle, the existing feature boolean is too blunt a proxy for when human close is actually needed. A better policy should likely consider richer signals instead of a single `feature` flag.

## What the doc must specify
1. Why `feature: true` is too coarse as the long-term human-close mechanism
2. What responsibilities blockers remove from human-close by shifting clarification earlier in the lifecycle
3. Candidate heuristic/policy inputs for human-close, such as:
   - unresolved blockers
n   - child completion/readiness
   - review requirements or explicit review state
   - evidence/verify quality or completeness
   - feature/goal scope or impact
   - risky side effects or operator-signoff requirements
4. How human-close remains distinct from blocker-based pre-decomposition / dispatch gating
5. Whether the future model should be a heuristic, explicit policy config, typed review requirement, or layered combination
6. How the migration could preserve compatibility with current `feature: true` repos while moving to a better model

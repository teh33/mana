  # Mana SQLite derived index — canonical implementation plan

  ## Summary
  Mana keeps canonical storage as human-editable typed Markdown+YAML files. SQLite is a derived, rebuildable index for fast/richer agent context assembly. Flow: human edit or mana command -> canonical `.mana/*.md` YAML frontmatter + Markdown body -> parse + validate typed fields -> derived SQLite -> stable mana-core APIs -> agent context/model input. SQLite is not source of truth; if missing/stale/corrupt, rebuild from canonical files.

  ## Settled decisions
  - Canonical source remains human-readable/editable `.mana/*.md`.
  - Typed YAML/frontmatter fields carry scheduler/agent semantics.
  - Markdown prose is first-class human context but not authority for machine state.
  - SQLite is derived/rebuildable and must not override canonical files.
  - JSONL is out of scope.
  - First SQLite value is agent context assembly, not generic list/show performance.
  - Mana APIs may read SQLite internally when fresh/valid.
  - Agents/external callers must not depend on raw SQLite tables.
  - Every mana command cheaply checks index health/freshness.
  - Every mana-controlled write validates candidate canonical content before landing, then updates SQLite.
  - Manual edits are detected on next mana command via freshness metadata.
  - Invalid YAML/frontmatter/schema blocks mana-controlled writes and blocks agent/scheduler use of relevant invalid units.
  - Existing `mana doctor` is extended for parse/schema and SQLite diagnostics.

  ## Current code boundary
  - Public/core API: `mana/crates/mana-core/src/api/mod.rs` (`create_unit`, `update_unit`, `close_unit`, `get_unit`, `list_units`, `load_index`, `assemble_context`, `get_status`, `get_tree`, dependency/ready-queue helpers).
  - Context surfaces: `ops/context.rs::assemble_agent_context(mana_dir, id)`, `ops/memory_context.rs::memory_context(mana_dir)`, prompt assembly paths that load units/context.
  - Current file index: `index.rs::Index`, `IndexEntry`, `Index::build/load/load_or_rebuild/is_stale`, `LockedIndex`.
  - Doctor: `mana-cli/src/commands/doctor.rs`; currently checks stale index, mixed formats, duplicate IDs, orphaned deps, missing parents, archived parent refs, stale/missing index entries, cycles, config findings; helper scans can skip unparsable unit files via `if let Ok(unit) = Unit::from_file`.
  - SQLite must be internal to mana-core, replacing/augmenting index/context internals, not becoming public DB API.

  ## API boundary
  Stable caller-facing APIs remain `api::assemble_context`, `ops::context::assemble_agent_context`, `ops::memory_context::memory_context`, `api::list_units`, `api::get_unit`, `api::load_index`. First SQLite slice moves only context assembly; others can stay file-index backed.

  Add internal module like `mana-core/src/derived_index/{mod,schema,rebuild,freshness,query}.rs`. Suggested internal API: `DerivedIndex::open`, `health`, `rebuild`, `update_unit_from_file`, `mark_stale`. Context assembly should call query helpers like `query_agent_context`, `query_memory_context`, `query_relevant_facts`, `query_dependency_providers`, `query_child_summaries`; return Rust structs owned by mana-core, never raw SQL/table access.

  ## SQLite file location
  Use derived project-local state, likely `.mana/index.sqlite`. It coexists with existing file index during migration. It can be deleted/rebuilt and should not be required for correctness or committed for correctness.

  ## Freshness contract
  Every SQLite read needs freshness check. Metadata: schema version, mana/index build version, mana root, last full rebuild timestamp, last incremental update timestamp, global stale flag/reason, source path/hash/mtime/size, parse/validation status. SQLite read allowed only when schema supported, not stale/corrupt, relevant source files match metadata, and no relevant unit invalid. Otherwise rebuild/update/fallback/block depending on command.

  ## Command lifecycle
  Every command performs cheap health check: DB exists, schema supported, stale flag, root metadata. Read-heavy commands check freshness, rebuild/incrementally refresh when safe, query stable APIs, block relevant invalid units, fallback to canonical scan only where explicitly permitted. Write commands (create/update/close/reopen/claim/release/facts/deps/attempts/delete/archive) flow: load canonical, mutate in memory, serialize candidate, parse through normal Unit parser, validate schema/invariants, atomic write, update affected SQLite rows, mark stale if SQLite update fails. Failed YAML/frontmatter/schema validation prevents canonical write from landing. Manual edits detected next command by hash/mtime/size; valid edits refresh SQLite, invalid edits report diagnostics and block relevant agent/scheduler use. No daemon/watcher v1.

  ## Invalid file handling
  Invalid canonical files must be explicit diagnostics, not silently skipped. SQLite records invalid files in diagnostic/source tables. Invalid files do not appear as valid unit rows. If previously valid unit becomes invalid, remove/mark stale old valid rows so agents cannot consume stale semantic state. Context: requested invalid unit fails; relevant dep/parent/child/fact invalid fails; unrelated invalid files may allow valid context plus diagnostics. Scheduler stricter than list/show.

  ## Proposed SQLite schema
  Fuller than current `IndexEntry`, optimized for context assembly.
  - `index_meta(key PRIMARY KEY, value)` for `schema_version`, `mana_version`, `mana_root`, `last_full_rebuild_at`, `last_incremental_update_at`, `stale`, `stale_reason`.
  - `source_files(path PRIMARY KEY, unit_id, kind, hash, mtime, size, indexed_at, status, error_kind, error_message, error_field)`, statuses `valid`, `invalid_parse`, `invalid_schema`, `missing`, `stale`, `archived`.
  - `units(id PRIMARY KEY, title, slug, status, priority, kind, unit_type, feature, created_at, updated_at, closed_at, close_reason, description, acceptance, notes, design, parent, assignee, claimed_by, claimed_at, is_archived, verify, verify_fast, fail_first, checkpoint, verify_hash, attempts, max_attempts, max_loops, verify_timeout, last_verified, stale_after, created_by, model, autonomy_disposition, outputs_json, on_fail_json, on_close_json, source_path, source_hash, indexed_at)`.
  - `unit_labels(unit_id, label)`.
  - `unit_paths(unit_id, path)`.
  - `unit_dependencies(unit_id, dep_id, position)`; avoid hard FK to dep_id unless missing/archived refs represented for doctor.
  - `unit_artifacts(unit_id, direction, artifact, position)` where direction is `produces`/`requires`.
  - `unit_decisions(unit_id, decision_index, text, resolved)`.
  - `unit_attempts(unit_id, attempt_index, num, outcome, notes, raw_json)`.
  - `unit_history(unit_id, history_index, started_at, finished_at, status, exit_code, raw_json)`.
  - `facts(unit_id PRIMARY KEY, last_verified, stale_after, score_hint)` for fact units/context filters.
  - Optional `context_edges(from_unit_id, to_unit_id, edge_type, weight, reason)` for parent/child/dependency/requires-produces/shared-path/shared-label edges; may defer and compute in Rust.
  - `index_diagnostics(id, severity, kind, source_path, unit_id, field, message, created_at)`.
  - FTS optional; sources: title, description, acceptance, notes, design, decisions, attempt notes. Defer unless needed for context assembly.

  ## Context assembly query flow
  `assemble_agent_context(mana_dir, id)`: open derived index -> check health/freshness -> rebuild/update if stale/safe -> load requested unit -> fail if invalid/missing -> load child summaries by parent -> load dependency providers from dependencies/artifacts -> load attempt notes -> load relevant facts from facts/paths/labels/text -> load referenced file contents from filesystem as today -> return existing `AgentContext` or stable equivalent. Preserve external `AgentContext` shape if possible. `memory_context` similarly queries active/claimed/in-progress units, recent updated/closed units, verified/non-stale facts, failed attempts/last failures.

  ## Doctor integration
  Extend existing `mana doctor`. New checks: SQLite missing/unsupported/stale/corrupt, source hash/mtime mismatch, parse/schema errors, unit rows missing for valid files, unit rows present for invalid/missing files, stale diagnostics older than current source hash. `doctor fix` may rebuild SQLite, remove stale derived rows, clear obsolete diagnostics. It must not silently rewrite canonical files, change status, delete files, resolve decisions, close units, or alter deps.

  ## Validation contract
  For mana-controlled writes: build candidate, serialize, parse via `Unit::from_string`, validate required invariants, only then write/replace. For manual edits: detect source metadata changes, parse/validate before use, block relevant invalid units, report file/field diagnostics. Classes: parse error, schema error, referential error, policy blocker, stale index warning. Parse/schema block relevant units; referential may block scheduler while still visible in list/doctor.

  ## Migration phases
  1. Schema/freshness metadata and internal module boundary.
  2. Full rebuild scanning canonical unit files with parse/validation diagnostics.
  3. Write-path integration with candidate validation and post-write index updates.
  4. Context assembly reads via SQLite-backed stable APIs.
  5. Existing doctor extended for validation/index repair.
  6. Later move list/search/dependency graph/ready queue/status/fact lookup to SQLite.

  ## Non-goals
  Do not make SQLite canonical. Do not add JSONL. Do not add daemon/watchers/server. Do not make Markdown read-only. Do not expose raw SQLite to agents. Do not silently repair/rewrite human-authored files. Do not solve numbering/identity here.

  ## Open implementation questions
  Ask before adding a new SQLite dependency if none exists or dependency choice is consequential. Decide during implementation: sync vs command-scoped background post-write updates, FTS now vs later, materialized context_edges vs Rust-computed, and how long existing file Index remains during transition.

  # Mana human-edit validation and doctor workflow — canonical implementation plan

  ## Summary
  Mana canonical files are edited by humans and consumed by agents. Human editability is first-class, but invalid typed mana files must block agent/scheduler use until repaired. Existing `mana doctor` is the main diagnostic/repair surface and should be extended for YAML/frontmatter parse errors, schema errors, and SQLite-derived index drift.

  ## Goals
  - Preserve direct human editing of `.mana/*.md`.
  - Prevent invalid YAML/frontmatter from becoming durable state through mana-controlled writes.
  - Detect manual edit breakage on the next mana command.
  - Block agents/scheduler from consuming invalid relevant units.
  - Report exact/actionable diagnostics: file, field if available, error kind, safe next action.
  - Extend existing `mana doctor`/`doctor fix`, not a parallel command.
  - Keep SQLite subordinate to canonical files.

  ## Current behavior
  `mana doctor` lives at `mana/crates/mana-cli/src/commands/doctor.rs`. It checks stale file index, mixed formats, duplicate IDs, orphaned deps, missing parents, archived parent refs, stale/missing index entries, cycles, config findings. Current scans may silently skip unparsable unit files using `if let Ok(unit) = Unit::from_file(&path)`, which is unsafe once agents rely on mana for context/scheduling.

  ## Validation classes
  - Parse errors: invalid YAML frontmatter, missing frontmatter delimiter, malformed envelope, shapes that prevent deserialization. Policy: block mana-controlled write before landing, block relevant agent/scheduler use, record doctor diagnostic, do not skip.
  - Schema errors: invalid status/kind/priority, non-string labels/paths/deps, bad timestamps, malformed typed actions. Same blocking policy as parse errors.
  - Referential errors: missing dep/parent, unexpected archived dep, cycles. Report in doctor; usually block scheduler for affected units; may allow list/show with diagnostics.
  - Policy blockers: unresolved decisions, human-only close, review requirements, claim/lease state. Typed action blockers, not parse/schema invalidity.
  - Derived-index warnings: SQLite missing/stale/unsupported, source hash mismatch, stale rows. Rebuild/refresh when safe; block only if canonical invalid or rebuild fails.

  ## Mana-controlled write contract
  Every command that writes unit files should use candidate-write flow: load current units, mutate in memory, serialize candidate canonical content, parse candidate with normal `Unit::from_string`, validate schema/invariants, atomic temp-file + rename write, update derived indexes after canonical write succeeds. If parse/schema validation fails, write must not land. Applies to create/update/close/reopen/claim/release/deps/facts/notes/decisions/attempts/archive/delete. Centralize around `Unit::to_file` or shared write helper; avoid per-command duplicate validation.

  ## Manual edit contract
  Humans may edit `.mana/*.md` directly. Next mana command detects changed path/hash/mtime/size. If file parses/validates, refresh derived state. If invalid, record diagnostic, show exact path and field/location when possible, block agent/scheduler use of relevant unit/context, do not auto-rewrite file.

  ## Context and scheduler blocking
  Requested invalid unit fails. Relevant invalid dep/parent/child/fact fails. Unrelated invalid file can allow valid context with diagnostics. Scheduler/run is strict: invalid target or dependency blocks; unrelated invalid files should be reported but need not block all execution. This prevents one unrelated broken file from making graph unusable while preventing corrupted context.

  ## Doctor behavior
  `mana doctor` should report parse/schema/referential/policy/index/config diagnostics. Diagnostic fields: severity, kind, path, unit id if known, field if known, message, suggested action. `doctor fix` may rebuild SQLite/file index, remove stale derived rows, clear diagnostics that no longer match current hashes, repair purely derived metadata. It must not silently rewrite canonical YAML, change statuses, delete units, resolve decisions, close units, or alter deps. Auto-fixing canonical files would require explicit separate design/flag.

  ## SQLite interaction
  Valid files populate `units` and child tables. Invalid files populate `source_files` / diagnostics with invalid status. Stale rows from previously valid versions are removed/marked invalid. Context queries check diagnostics/relevance before returning. Derived SQLite never overrides canonical parse errors.

  ## Integration points
  Inspect/modify: `unit/mod.rs` (`from_file`, `from_string`, `to_file`, validation helpers), `index.rs` (stale/rebuild parse error handling), `api/mod.rs` (write API/context/list/index behavior), `ops/*` write operations, `ops/context.rs`, `ops/memory_context.rs`, `mana-cli/src/commands/doctor.rs`.

  ## Suggested validation API
  Create shared validation layer used by doctor, SQLite rebuild, write paths, and context assembly. Suggested shapes: `ValidationDiagnostic { severity, kind, path, unit_id, field, message, suggested_action }`, `ValidationKind::{Parse, Schema, Referential, Policy, Index}`, `validate_unit_file(path) -> ValidationResult<Unit>`, `validate_unit_candidate(content) -> ValidationResult<Unit>`, `validate_graph(mana_dir) -> Vec<ValidationDiagnostic>`.

  ## Migration phases
  1. Make parse/schema diagnostics explicit; stop silently skipping unparsable files in doctor scans.
  2. Protect mana-controlled writes with serialize -> parse -> validate -> atomic write.
  3. Integrate with SQLite: record invalid sources/diagnostics, remove stale valid rows, block invalid relevant context.
  4. Improve repair UX: precise file/field diagnostics and safe `doctor fix` rebuilds.

  ## Non-goals
  Do not make human editing read-only. Do not silently rewrite canonical files. Do not make SQLite canonical. Do not add a parallel doctor. Do not block all mana usage because one unrelated file is invalid. Do not solve numbering migration here.

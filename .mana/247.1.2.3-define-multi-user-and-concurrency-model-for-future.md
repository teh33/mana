Goal: Define the multi-user/shared-project implications of mana storage evolution.

Settled current model:
- Local/project mode keeps human-editable Markdown+YAML files canonical.
- Local `.mana/index.sqlite` is derived/rebuildable per checkout and should not be committed/treated as shared canonical state.
- Do not put a mutable SQLite DB file in Git and expect safe multi-user merges.

Forward hypothesis:
- A hosted/server SQLite or SQLite-backed service can become team authority later.
- The current SQLite schema/API boundary is useful as a stepping stone: the same typed query/write model could run server-side while local agents continue using stable mana APIs.
- Multi-user sync requires a separate authority/conflict design before implementation: auth, optimistic revisions, conflict visibility, leases, merge/apply behavior, and offline recovery.

Recommended staged path:
1. Keep local-first canonical Markdown+YAML now.
2. Keep SQLite derived locally for context/query performance.
3. Do not share local `.mana/index.sqlite` through Git as authority.
4. When multi-user pressure is concrete, design hosted authority over typed objects/API operations; SQLite may be the implementation store behind that service.
5. Preserve human editability by making local file edits explicit sync/apply operations into the hosted authority rather than transparent bidirectional magic.

Out of scope for current SQLite-derived index work:
- Hosted server implementation.
- Auth/permissions model.
- Real-time sync.
- Making SQLite canonical in local Git repos.

Goal: Define the multi-user/shared-project implications of mana storage evolution.

Current discussion context:
- User expects future projects with multiple users sharing mana.
- A central file/source of truth may be stronger than many Markdown files, but multi-user changes the storage and conflict model.
- Need to decide if mana remains local-first Git-synced, uses a shared project DB file, uses SQLite as local cache, or eventually has a service/server authority.

Steps:
1. Enumerate likely deployment modes: single-user local repo, team Git repo, shared filesystem, central service, and hosted product.
2. For each mode, evaluate current MD files, JSONL, SQLite DB file, and server-backed DB implications.
3. Identify locking/concurrency requirements: optimistic concurrency, object revisions, append-only event IDs, leases, conflict resolution, and backup/restore.
4. Recommend a staged model that does not overbuild a server today but does not block it later.
5. Capture user-facing guarantees: offline operation, merge behavior, conflict visibility, and recoverability.

Files:
- `docs/architecture/mana-storage-backend-evolution.md` (modify or linked section)
- optionally `docs/architecture/mana-multi-user-storage-model.md`

In scope:
- Architecture-level concurrency and sharing model.

Out of scope:
- Implementing server sync.
- Auth/permissions product design beyond storage implications.

Do not:
- Treat a single SQLite file in Git as automatically safe for concurrent team editing.
- Ignore offline/local-first workflows.
- Assume daemon/database dependency is acceptable for all mana users.

## Task
Write a root architecture doc that defines the future trigger model for explicit human-close as a separate close path from ordinary verify-close.

## Why this unit exists
The current discussion narrowed the human-close redesign significantly:
- verify-close remains fine and should remain the normal close path
- the redesign target is only the explicit human-close path currently triggered too coarsely by `feature: true`
- explicit review/policy state should drive explicit human-close when that path is requested or required

## What the doc must specify
1. The two close paths explicitly
   - ordinary verify-close
   - explicit human-close
2. What can request or require explicit human-close
   - explicit review requirement
   - explicit policy/config
   - possibly compatibility behavior from current `feature: true`
3. What does *not* automatically trigger explicit human-close
   - ordinary verify success/failure by itself
   - blockers by default
   - generic heuristics over all closes
4. How the trigger model preserves compatibility with current repos
5. How this trigger model relates to blockers and to future human-close policy work without replacing normal close behavior

## Do not
- do not redesign ordinary verify-close here
- do not make blockers automatically imply human-close
- do not frame human-close as a general close heuristic over all units

Goal: Produce a focused architecture comparison for mana's canonical storage backend.

Current state:
- Mana currently stores unit records in `.mana/*.md` with YAML frontmatter plus Markdown body, parsed into typed `Unit` structs.
- This is git-friendly and human-readable but unit-centric, mutable, and less ideal for future multi-user/shared-project coordination.
- User raised SQLite, JSONL, or another central source-of-truth as possible future directions.

Steps:
1. Inspect current mana storage paths and parsing/indexing code enough to ground the comparison: `mana/crates/mana-core/src/unit/mod.rs`, index handling, archive handling, and any config/index files.
2. Compare four options: current MD+YAML frontmatter, pure JSONL append log, SQLite database, and object-file graph with generated indexes.
3. Evaluate each against: git diffability, offline/local-first use, multi-user conflict behavior, query/index performance, schema migration, human editability, agent robustness, and implementation cost.
4. Recommend one near-term canonical direction and one migration-safe fallback.
5. Capture open questions that need user decision before implementation.

Files:
- `docs/architecture/mana-storage-backend-evolution.md` (create/modify — comparison and recommendation)
- `mana/crates/mana-core/src/unit/mod.rs` (read only)
- relevant index/archive code (read only)

In scope:
- Architecture comparison and recommendation.
- No code changes.

Out of scope:
- Implementing SQLite/JSONL.
- Migrating existing mana files.
- Network sync/server design beyond local storage implications.

Do not:
- Assume SQLite is automatically correct.
- Discard current Markdown-frontmatter compatibility.
- Treat human-editable prose as canonical scheduler state.

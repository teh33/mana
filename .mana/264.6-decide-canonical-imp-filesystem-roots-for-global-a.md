Goal: decide the canonical top-level filesystem roots imp should use, and resolve whether imp should standardize on an imp-native `~/.imp/` root or a platform-native XDG/Library split for all user-global storage.

Current state:
- The durable design question has narrowed to top-level roots, not just per-artifact categorization.
- Current imp behavior mixes multiple user-global roots (`~/.config/imp`, `~/.local/share/imp`, macOS Application Support, and some legacy `~/.imp`) plus project-local `<project>/.imp/`.
- This mixed model is the core source of confusion and should be resolved before lower-level path-helper refactors.

Decision to make:
- Option A: canonical user-global root is `~/.imp/`, with `<project>/.imp/` for project-local overrides; XDG/Library paths become compatibility/migration-only.
- Option B: canonical user-global root is fully platform-native (`~/.config/imp`, `~/.local/share/imp`, `~/Library/Application Support/imp` as appropriate), with `<project>/.imp/` for project-local overrides.

Steps:
1. Compare the two top-level root strategies against product legibility, operator usability, docs/support burden, packaging/platform convention, and migration cost.
2. Make an explicit decision on the canonical global root policy.
3. Define the exact subdirectory contract that follows from that decision.
4. Define the compatibility and migration policy for legacy/non-canonical roots.
5. Feed the result back into 264.2 and the implementation follow-ons.

Done when:
- imp has one explicit canonical user-global root policy and one project-local root policy
- compatibility-only roots are clearly distinguished from source-of-truth roots
- the result is concrete enough to guide docs and implementation

Out of scope:
- implementing the migration
- broader session-index lifecycle wiring except as it affects root choice

Goal: Specify the policy for Markdown files if mana moves to a stronger canonical storage backend.

Current discussion context:
- User suggested Markdown could act as a mirror for a DB, with changes reflected DB <-> MD.
- This is consequential because bidirectional sync can create split-brain state, conflict complexity, and ambiguous authority.
- Current mana already treats Markdown body mostly as human description while typed frontmatter is parsed as Unit state.

Steps:
1. Define three policy options: generated read-only Markdown mirror; editable Markdown with explicit import/sync command; fully bidirectional live sync.
2. For each option, describe conflict/drift behavior, expected user workflow, and implementation risk.
3. Recommend a near-term policy that protects canonical truth while preserving human editability where valuable.
4. Define required metadata if sync is supported: source revision/hash, last materialized timestamp, object IDs, dirty state, conflict markers, and validation errors.
5. State which fields may be edited in Markdown safely and which typed state must go through mana APIs/commands.

Files:
- `docs/architecture/mana-storage-backend-evolution.md` (modify or linked section)
- optionally `docs/architecture/mana-markdown-mirror-policy.md` (create only if the storage doc gets too large)

In scope:
- Markdown mirror/editability policy.
- Sync conflict model at architecture level.

Out of scope:
- Implementing watchers or sync commands.
- UI design for conflict resolution.

Do not:
- Promise transparent bidirectional sync without conflict semantics.
- Make Markdown prose authoritative for scheduler behavior.
- Remove current `.mana/*.md` compatibility in the first migration slice.

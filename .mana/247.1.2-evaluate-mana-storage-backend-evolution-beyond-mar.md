  Durable decomposition from 2026-04-27 storage discussion:

  Target layered model:
  1. Canonical typed objects/events: Task, Attempt, Decision, Blocker, Evidence, Claim/Assertion, Review, Artifact, Receipt/Event. This is the authority for scheduler/agent semantics.
  2. Durable object/event log: likely JSONL or object files, append-friendly and git/replay/recovery friendly. If SQLite and Markdown disappear, mana should rebuild from these canonical records.
  3. Query/index store: SQLite is valuable for fast lookup, graph/scheduler queries, filtering, and future multi-user views, but starts as materialized/rebuildable index/cache.
  4. Human mirror: Markdown remains a generated view and optional explicit import surface. DB/object store -> Markdown can be automatic; Markdown -> canonical store should be explicit `mana import/sync/apply` with validation, source hashes, dirty/conflict state, and field-level policy.

  Near-term recommendation: do not make SQLite-only canonical yet. First specify object/event schema and invariants; then add SQLite as rebuildable index; then define Markdown mirror policy; then design multi-user/server authority as a later layer over the same event/object substrate.

  Numbering direction: decouple canonical identity from display path. Existing `x.y.z` remains compatibility/human display, but future canonical references should use stable opaque object IDs with parent/order/display metadata. This avoids breaking existing references while making insertion, reordering, sync, and long-lived references safer.

Goal: Evaluate and specify whether mana should evolve from Markdown-frontmatter unit files as canonical storage to a typed storage backend such as SQLite or JSONL, with Markdown as a mirror/view.

Current discussion context:
- Mana currently parses `.mana/*.md` as YAML frontmatter plus optional Markdown body and stores scheduler-relevant fields as typed YAML, not by parsing Markdown prose.
- User is considering future multi-user/shared-project mana where a central source of truth may be stronger than many mutable `.md` files.
- Candidate direction: SQLite or JSONL/event-log/object store as canonical source of truth; Markdown files become deterministic mirrors for human editing/review. Changes may need bidirectional sync: DB -> Markdown and Markdown -> DB.
- Related design pressure from Markdown critique: Markdown should be human authoring/rendering, not canonical programmable substrate.

Questions to answer:
1. What backend should be canonical first: SQLite, JSONL append log, object files, or hybrid?
2. Should Markdown remain editable, or become read-only/generated mirror by default?
3. If bidirectional sync is supported, what conflict model prevents DB/Markdown drift?
4. How does this support multi-user/shared-project workflows without introducing daemon/database operational burden too early?
5. How does this relate to existing file-native object-memory direction under 247.1?

Deliverable:
- A concise architecture decision/spec comparing SQLite, JSONL, and current MD+YAML frontmatter.
- Recommended near-term proving slice that preserves current git-friendly workflow while enabling stronger centralized/indexed storage later.

Out of scope:
- Immediate database migration.
- Networked server/multi-tenant service implementation.
- Breaking current `.mana/*.md` compatibility.

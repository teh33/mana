Goal: implement a central imp-core storage-path API and migrate existing callers to it with minimal behavior drift, after the storage contract is specified.

Current state:
- Callers currently resolve paths ad hoc across config, resources, session search, memory, extension management, import, and runtime flows.
- We need one shared source of truth before fixing the remaining path mismatches.

Dependencies:
- depends on 264.1 audit
- depends on 264.2 normalized storage contract

Steps:
1. Add a central storage-path module/helper in `imp-core` that exposes named helpers for each durable storage surface.
2. Migrate the highest-risk callers first: sessions, session search index, auth/memory, resources/skills/prompts/soul, Lua discovery, shell-tool discovery/import paths.
3. Preserve compatibility lookup where the spec requires it.
4. Add focused tests for platform/path behavior instead of relying only on callsite tests.
5. Leave docs updates to the dedicated follow-on job unless a code comment is required for clarity.

Files:
- `imp/crates/imp-core/src/config.rs` or a new nearby module if a separate storage-path module is cleaner
- `imp/crates/imp-core/src/resources.rs`
- `imp/crates/imp-core/src/tools/session_search.rs`
- `imp/crates/imp-core/src/tools/memory.rs`
- `imp/crates/imp-core/src/tools/extend.rs`
- `imp/crates/imp-core/src/tools/web/*`
- `imp/crates/imp-core/src/import.rs`
- `imp/crates/imp-lua/src/*`
- related tests

Done when:
- current callers stop open-coding the main durable imp paths
- the highest-risk path mismatch behavior is covered by focused tests

Do not:
- broaden into unrelated config redesign
- silently strand legacy user data without an intentional compatibility path
